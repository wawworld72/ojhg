# Feature Specification: Google Classroom-Integrated Online Judge

**Feature Branch**: `001-online-judge-classroom`
**Created**: 2026-05-11
**Updated**: 2026-05-11
**Status**: Draft (v2 — 문제 세트 구조, 시도 횟수 기반 채점, Classroom 예약/마감 연동 반영)

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 학생: Classroom 과제에서 문제 세트 시작 및 코드 제출 (Priority: P1)

학생이 Google Classroom에서 프로그래밍 과제를 열면 온라인 저지의 **문제 세트 페이지**로 이동한다.
문제 세트에는 교사가 구성한 여러 개의 문제가 포함되어 있으며, 학생은 각 문제에 코드를 제출한다.
각 문제는 시도 횟수에 따라 획득 가능한 점수가 다르며, 모든 문제의 점수 합산이 최종 성적으로 Google Classroom에 반영된다.

**Why this priority**: 시스템의 핵심 사용자 흐름. 이것 없이 다른 기능이 의미 없다.

**Independent Test**: 학생 계정으로 Classroom 과제 클릭 → 문제 세트 페이지 진입 → 문제 1 코드 제출 → 채점 결과 + 획득 점수 확인.

**Acceptance Scenarios**:

1. **Given** 과제가 공개 기간 중이고 로그인한 학생이 Classroom 과제 링크를 클릭했을 때, **When** 페이지가 로드되면, **Then** 문제 세트 목록(문제 번호, 제목, 배점, 현재 획득 점수, 남은 시도 횟수 또는 현재 획득 가능 점수 구간)이 표시된다.
2. **Given** 학생이 특정 문제에 코드를 제출했을 때, **When** 채점이 완료되면, **Then** 각 테스트 케이스 결과와 함께 이번 제출로 획득한 점수, 누적 시도 횟수가 표시된다.
3. **Given** 학생이 5번 이내 시도에서 Accepted를 받으면, **When** 결과가 표시되면, **Then** 해당 문제의 만점이 적용된다.
4. **Given** 학생이 6~10번 시도에서 처음으로 Accepted를 받으면, **When** 결과가 표시되면, **Then** 교사가 설정한 해당 구간 점수(예: 90%)가 적용된다.
5. **Given** 과제 마감 시간이 지난 후 학생이 제출하면, **When** 제출 버튼을 누르면, **Then** 마감 이후 제출임이 표시되고 교사 설정에 따라 허용 또는 차단된다.

### User Story 2 - 교사: 문제 세트 구성 및 Classroom 과제 연동 (Priority: P1)

교사가 온라인 저지에서 문제 세트를 만들고, 그 안에 여러 문제를 추가한다.
각 문제에는 테스트 케이스와 **시도 횟수별 점수 구간**을 설정한다.
완성된 문제 세트를 Google Classroom 과제와 연결하면, Classroom의 예약 공개/마감 설정이 온라인 저지에도 자동으로 적용된다.

**Why this priority**: 교사가 문제 세트를 만들어야 학생 스토리가 가능하다.

**Independent Test**: 교사가 2개 문제 포함 문제 세트 생성 → Classroom 과제 연동 → 학생 계정으로 해당 과제 접근 확인.

**Acceptance Scenarios**:

1. **Given** 교사가 문제 세트 생성 화면에서 문제를 추가하면, **When** 문제 제목, 설명, 테스트 케이스, 시도 횟수별 점수 구간, 배점, 허용 언어를 입력하고 저장하면, **Then** 해당 문제가 문제 세트에 포함된다.
2. **Given** 교사가 시도 횟수별 점수 구간을 설정할 때, **When** 구간을 정의(예: 1~5회 100%, 6~10회 80%, 11~20회 60%, 21회 이상 0%)하고 저장하면, **Then** 학생이 Accepted를 받은 시점의 누적 시도 횟수에 따라 해당 구간 점수가 적용된다.
3. **Given** 교사가 Google Classroom 과제를 선택하여 문제 세트와 연동하면, **When** 연동이 완료되면, **Then** Classroom에 설정된 공개 예약 시각과 마감 시각이 온라인 저지에 자동으로 동기화된다.
4. **Given** Classroom 과제의 마감 시각이 변경되면, **When** 다음 동기화 시, **Then** 온라인 저지의 해당 문제 세트 마감 시각도 갱신된다.
5. **Given** 교사가 문제 세트를 저장하면, **When** 저장이 완료되면, **Then** 해당 문제 세트로 연결되는 공유 링크가 생성된다.

### User Story 3 - Google Classroom 최종 성적 자동 반영 (Priority: P1)

학생이 문제 세트 내 문제들을 제출할 때마다 각 문제의 획득 점수가 업데이트되고, 전체 문제 세트의 합산 점수가 Google Classroom 성적부에 자동으로 반영된다.

**Why this priority**: 구글 클래스룸 연동이 핵심 가치이므로 P1으로 상향.

**Independent Test**: 학생이 문제 세트 내 모든 문제 제출 → 각 문제 점수 합산 → Google Classroom 성적부에서 최종 점수 확인.

**Acceptance Scenarios**:

1. **Given** 학생이 문제 세트의 문제 1을 Accepted로 완료하면, **When** 채점이 완료되면, **Then** 문제 1의 점수가 문제 세트 합계에 반영되고 Google Classroom 성적부가 5분 이내에 갱신된다.
2. **Given** 학생이 동일 문제를 재제출하여 더 낮은 점수를 받으면, **When** Classroom에 반영될 때, **Then** 해당 문제의 최고 점수가 유지되고 합산 점수가 보존된다.
3. **Given** 문제 세트 마감 시간이 지나면, **When** 마감이 도래하면, **Then** 모든 학생의 현재 합산 점수가 최종 확정되어 Google Classroom에 일괄 반영된다.
4. **Given** Google Classroom API 호출이 실패하면, **When** 재시도가 모두 소진되면, **Then** 실패 기록이 저장되고 교사에게 알림이 전달된다.

---

### User Story 4 - 교사: 과제 현황 모니터링 (Priority: P2)

교사가 문제 세트별로 학생들의 진행 현황(문제별 제출 수, Accepted 비율, 평균 시도 횟수, 평균 점수)을 볼 수 있다.

**Why this priority**: 편의 기능, MVP 이후 추가 가능.

**Independent Test**: 교사 대시보드 → 문제 세트 통계 페이지 → 문제별 학생 현황 확인.

**Acceptance Scenarios**:

1. **Given** 교사가 문제 세트 통계 페이지를 열면, **When** 로드되면, **Then** 문제별로 (전체 학생 수, 제출 완료 수, Accepted 수, 평균 시도 횟수, 평균 획득 점수)가 표시된다.
2. **Given** 교사가 특정 학생의 이름을 클릭하면, **When** 상세 페이지가 열리면, **Then** 해당 학생의 모든 문제 제출 이력(시각, 시도 횟수, 결과, 코드, 획득 점수)을 볼 수 있다.

### Edge Cases

- 학생이 Classroom 수업에 미등록된 상태에서 저지 링크에 직접 접근하면? → 접근 차단, 해당 수업 등록 안내.
- 과제 공개 예약 시각 이전에 학생이 링크를 열면? → "과제가 아직 공개되지 않았습니다"와 공개 시각 표시.
- 교사가 마감 후 특정 학생에게 연장 허용하면? → Classroom의 과제 개별 마감 연장 기능을 온라인 저지에도 동기화.
- 문제 세트 내 일부 문제만 제출한 상태에서 마감되면? → 미제출 문제는 0점 처리, 제출된 문제의 최고 점수 합산이 최종 성적.
- 동일 학생이 동시에 같은 문제를 두 탭에서 제출하면? → 먼저 채점 완료된 결과만 시도 횟수에 반영, 중복 카운트 방지.
- 교사가 이미 제출이 있는 문제의 시도 횟수 구간을 수정하면? → 기존 제출 결과에는 이전 구간 기준을 유지, 이후 제출부터 새 구간 적용, 교사에게 소급 적용 여부 선택 제공.
- 채점 중 학생이 브라우저를 닫으면? → 서버에서 채점 계속 진행, 학생이 재접속 시 결과 확인 가능.
- Classroom 과제가 삭제되면? → 온라인 저지 내 문제 세트는 보존(학생 제출 이력 보호), 교사에게 연동 해제 알림.
- 교사가 마감 전에 Publish를 시도하면? → 차단, "마감 후에만 게시 가능합니다" 안내.
- 동일 과제를 재 Publish하면? → 기존 학생 브랜치에 신규 커밋을 추가(덮어쓰지 않음); 이미 업로드된 커밋은 유지.
- 학생이 제출 시 익명 처리를 원하면? → v1에서 익명화는 지원하지 않음; 교사가 저장소를 private으로 관리하는 것을 권장.
- GitHub 저장소가 존재하지 않으면? → 시스템이 교사 계정에 자동으로 저장소를 생성(private 기본값).
- 학생이 GitHub 계정이 없어도? → 학생 식별자는 Google 이메일 앞부분(@ 이전)을 슬러그로 사용; 학생 GitHub 계정 불필요.

## Requirements *(mandatory)*

### Functional Requirements

**인증 & 접근 제어**
- **FR-001**: 시스템은 Google OAuth 2.0만을 통한 로그인을 허용해야 한다.
- **FR-002**: 시스템은 Google Classroom API를 통해 사용자의 역할(교사/학생)과 수업 소속을 확인하여 접근 권한을 부여해야 한다.
- **FR-003**: 학생은 자신이 등록된 수업의 과제에 연결된 문제 세트에만 접근할 수 있어야 한다.

**Classroom 연동 & 일정 동기화**
- **FR-004**: 시스템은 Google Classroom 과제의 예약 공개 시각(scheduled time), 마감 시각(due date/time), 점수 배점(max points)을 자동으로 가져와 문제 세트에 적용해야 한다.
- **FR-005**: Google Classroom 과제 일정이 변경되면 시스템은 1시간 이내에 해당 문제 세트의 일정을 동기화해야 한다.
- **FR-006**: 과제 공개 시각 이전에는 학생이 문제 세트에 접근할 수 없어야 한다.
- **FR-007**: 교사는 마감 이후 제출의 허용 여부를 문제 세트별로 설정할 수 있어야 한다. 마감 이후 제출을 허용하는 경우 "지각 제출" 여부가 표시되어야 한다.
- **FR-008**: Classroom의 개별 학생 마감 연장(extension)이 설정된 경우, 해당 학생의 온라인 저지 접근 마감도 동기화되어야 한다.

**문제 세트 관리 (교사)**
- **FR-009**: 교사는 문제 세트를 생성하고 그 안에 1개 이상의 문제를 추가할 수 있어야 한다.
- **FR-010**: 교사는 문제 세트를 Google Classroom의 특정 과제에 연결할 수 있어야 한다. 하나의 Classroom 과제에는 하나의 문제 세트만 연결된다.
- **FR-011**: 교사는 문제 세트 내 문제의 순서를 변경할 수 있어야 한다.

**문제 관리 (교사)**
- **FR-012**: 교사는 문제 제목, 설명, 입출력 예시, 시간 제한(초), 메모리 제한(MB), 배점, 허용 프로그래밍 언어를 설정하여 문제를 생성할 수 있어야 한다.
- **FR-013**: 교사는 문제당 최소 1개, 최대 100개의 테스트 케이스(입력 파일 + 예상 출력 파일)를 업로드할 수 있어야 한다.
- **FR-014**: 교사는 각 테스트 케이스를 공개(학생에게 입출력 노출) 또는 비공개(결과만 노출)로 설정할 수 있어야 한다.
- **FR-015**: 교사는 문제별로 시도 횟수 기반 점수 구간을 1개 이상 설정할 수 있어야 한다. 각 구간은 (최소 시도 횟수, 최대 시도 횟수, 점수 비율%)로 구성된다. 어떤 구간에도 해당하지 않으면 0점으로 처리된다.
- **FR-016**: 교사는 이미 학생 제출이 존재하는 문제의 점수 구간을 수정할 경우, 기존 제출에 소급 적용할지 여부를 선택할 수 있어야 한다.

**코드 제출 & 채점 (학생)**
- **FR-017**: 학생은 허용된 프로그래밍 언어 중 하나를 선택하고, 브라우저 내 코드 에디터(문법 강조, 자동 들여쓰기 지원)에 코드를 직접 작성하거나 파일(.py/.java/.cpp/.c)을 업로드하여 제출할 수 있어야 한다.
- **FR-018**: 시스템은 제출된 코드를 격리된 환경에서 실행하고 각 테스트 케이스에 대해 Accepted / Wrong Answer / Time Limit Exceeded / Memory Limit Exceeded / Runtime Error / Compilation Error 중 하나의 결과를 반환해야 한다.
- **FR-019**: 채점 결과는 제출 후 30초 이내에 학생에게 표시되어야 한다 (코드 500줄, 테스트 케이스 20개 기준).
- **FR-020**: 시스템은 학생이 해당 문제에서 시도한 누적 횟수를 추적하고, 문제 페이지에 현재 시도 횟수와 획득 가능한 점수 구간을 표시해야 한다.
- **FR-021**: 학생이 Accepted를 처음 받는 시점의 누적 시도 횟수를 기준으로 해당 점수 구간의 점수가 적용된다. 이미 Accepted를 받은 문제를 재제출하더라도 점수는 변경되지 않는다.
- **FR-022**: 학생은 자신의 모든 제출 이력(시각, 언어, 결과, 시도 횟수, 획득 점수)을 조회할 수 있어야 한다.
- **FR-023**: 학생은 과거 제출의 코드를 재열람할 수 있어야 한다.

**성적 산출 및 Classroom 반영**
- **FR-024**: 문제 세트의 최종 성적은 세트 내 모든 문제의 획득 점수 합산으로 계산된다.
- **FR-025**: 각 문제 제출 후 채점이 완료되면 시스템은 해당 학생의 문제 세트 합산 점수를 Google Classroom 성적부에 5분 이내에 반영해야 한다.
- **FR-026**: Google Classroom 성적 반영은 멱등적이어야 한다.
- **FR-027**: 문제 세트 마감 시각 도래 시, 미제출 문제는 0점 처리되고 각 학생의 최종 합산 점수가 Google Classroom에 일괄 확정 반영되어야 한다.
- **FR-028**: Google Classroom API 호출 실패 시 시스템은 최대 3회 재시도 후 실패 이력을 기록하고 교사에게 알림을 전달해야 한다.

**제출 기록 보존**
- **FR-029**: 학생의 모든 제출(코드, 언어, 제출 시각, 시도 횟수, 채점 결과, 각 테스트 케이스 결과)은 과제 마감 이후에도 영구적으로 보존되어야 한다.
- **FR-030**: 학생은 자신의 문제별 전체 제출 이력을 제출 순서대로 조회할 수 있어야 한다. 각 제출의 코드를 열람할 수 있어야 한다.

**교사 대시보드 — 수업 · 과제 · 학생별 정리**
- **FR-031**: 교사는 자신이 개설한 여러 Google Classroom 수업을 하나의 대시보드에서 조회할 수 있어야 한다.
- **FR-032**: 교사는 수업(Course) → 과제(Assignment) → 학생(Student) 계층 구조로 제출 현황을 탐색할 수 있어야 한다.
- **FR-033**: 교사는 과제 단위로 전체 학생의 문제별 현황(제출 수, Accepted 수, 평균 시도 횟수, 평균 점수)을 조회할 수 있어야 한다.
- **FR-034**: 교사는 특정 학생을 선택하면 해당 학생의 과제 내 모든 문제에 대한 제출 이력 전체(시도 순서대로 정렬된 코드 + 채점 결과)를 조회할 수 있어야 한다.

**코드 유사도 분석 (부정행위 탐지)**
- **FR-035**: 시스템은 과제 마감 후 교사의 요청 또는 자동으로 해당 과제에 제출된 모든 학생의 코드를 문제별로 쌍(pair)을 이루어 유사도를 분석해야 한다.
- **FR-036**: 유사도 분석 결과는 학생 쌍(A-B)별 유사도 점수(0~100%)로 표시되어야 하며, 교사가 설정한 임계값(기본 80%) 이상인 쌍은 "주의 요망"으로 강조 표시되어야 한다.
- **FR-037**: 교사는 "주의 요망" 쌍을 클릭하면 두 학생의 코드를 나란히(diff 형태로) 비교할 수 있어야 한다.
- **FR-038**: 유사도 분석은 Accepted 제출뿐 아니라 모든 제출 이력을 대상으로 수행할 수 있어야 한다.

**GitHub 코드 게시 (Publish)**
- **FR-039**: 교사는 특정 과제(Assignment)를 선택하여 "Publish" 버튼을 실행할 수 있어야 한다. Publish는 과제 마감 후에만 실행 가능하다.
- **FR-040**: Publish 실행 시 시스템은 교사의 GitHub 계정에 연결된 저장소의 지정 클래스룸 폴더 하위에 해당 과제 폴더를 자동 생성해야 한다. 저장소 구조는 `{course-name}/assignments/{assignment-name}/` 이다.
- **FR-041**: Publish 시 각 학생의 모든 제출 시도가 **문제(Problem) 단위 전용 브랜치** (`submissions/{assignment-name}/{problem-slug}/{student-identifier}`)에 시도 순서대로 개별 커밋으로 업로드되어야 한다. 브랜치를 문제 단위로 분리함으로써 `git log`에서 단일 문제에 대한 순수한 풀이 이력만 추적할 수 있어야 한다. 커밋 메시지에는 시도 번호, 채점 결과, 제출 시각, 언어, 점수가 포함되어야 한다.
- **FR-042**: 동일 저장소에 50명의 학생 브랜치가 공존하더라도 각 브랜치는 독립적이므로 서로 영향을 주지 않는다. 브랜치 수는 학생 수 × 문제 수이며 GitHub은 이를 충분히 처리할 수 있어야 한다.
- **FR-043**: Publish는 비동기로 처리되어야 하며, 교사는 진행 상태(대기 중 / 업로드 중 N명 / 완료)를 실시간으로 확인할 수 있어야 한다.
- **FR-044**: Publish 완료 후 교사는 생성된 GitHub 저장소 URL 및 각 학생 브랜치 URL을 대시보드에서 조회할 수 있어야 한다.
- **FR-045**: GitHub 업로드 실패 시(API 오류, 권한 오류 등) 실패한 학생 목록과 오류 원인이 교사에게 표시되어야 하며 해당 학생만 재시도할 수 있어야 한다.

### Key Entities

- **ProblemSet**: 문제 세트 이름, 연결된 Classroom 과제 ID, 포함된 문제 목록(순서 포함), 공개 시각, 마감 시각, 마감 후 제출 허용 여부, 총 배점.
- **Problem**: 제목, 설명, 입출력 예시, 시간/메모리 제한, 배점, 허용 언어, 속한 ProblemSet, 시도 횟수별 점수 구간 목록.
- **AttemptScoreTier**: 최소 시도 횟수, 최대 시도 횟수, 점수 비율(%), 속한 Problem. (예: min=1, max=5, ratio=100 / min=6, max=10, ratio=80)
- **TestCase**: 입력 데이터, 예상 출력 데이터, 공개 여부, 속한 Problem.
- **Submission**: 제출 학생, 속한 Problem, 코드 내용, 언어, 제출 시각, 지각 제출 여부, 채점 결과, 각 테스트 케이스 결과, 누적 시도 횟수(해당 제출 시점), 획득 점수.
- **StudentProblemProgress**: 학생, Problem, 누적 시도 횟수, 첫 Accepted 시도 횟수, 최종 획득 점수.
- **ClassroomCourse**: Google Classroom 수업 ID, 수업명, 교사 목록, 학생 목록. (교사 한 명이 여러 수업 개설 가능)
- **ClassroomAssignment**: Google Classroom 과제 ID, 속한 Course, 연결된 ProblemSet, 배점, 예약 시각, 마감 시각.
- **User**: Google 계정 ID, 이름, 이메일. 역할은 수업별로 결정(CourseEnrollment).
- **SimilarityReport**: 과제, 문제, 학생 A, 학생 B, 유사도 점수(%), 비교 대상 제출 ID(A/B), 분석 시각, 플래그 여부(임계값 초과).
- **GitHubPublish**: 과제 ID, 교사 GitHub 연동 정보, 저장소명, 게시 상태(pending/running/completed/failed), 시작/완료 시각, 학생별 업로드 결과(성공/실패/브랜치 URL).
- **GitHubIntegration**: 교사 User ID, GitHub 계정명, OAuth 액세스 토큰(암호화), 대상 저장소명, 연동 시각.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 학생이 Classroom 과제 링크 클릭 후 3번 이내의 클릭으로 문제 세트의 첫 문제에 코드를 제출할 수 있다.
- **SC-002**: 채점 결과가 제출 후 **5분 이내**에 학생 화면에 표시된다 (피크 부하 시 대기 포함; 비피크 시 30초 이내).
- **SC-003**: Google Classroom 성적 반영이 채점 완료 후 5분 이내에 완료된다.
- **SC-004**: Classroom 과제 일정(공개/마감) 변경이 1시간 이내에 온라인 저지에 동기화된다.
- **SC-005**: 시스템은 1회 수업(50명 × 5문제 × 20시도 = 최대 5,000건/60분, 피크 2.8건/초)을 처리할 수 있다.
- **SC-006**: 악의적인 제출 코드(무한루프, 포크 폭탄, 파일시스템 접근)가 호스트 시스템에 영향을 미치지 않는다.
- **SC-007**: 교사가 문제 세트(문제 2개 포함)를 만들고 Classroom 과제와 연동하는 전체 과정을 15분 이내에 완료할 수 있다.
- **SC-008**: Google Classroom 성적 반영 성공률이 99% 이상이다 (재시도 포함).
- **SC-009**: 학생이 문제 페이지에서 현재 시도 횟수와 다음 제출 시 획득 가능한 점수 구간을 즉시 확인할 수 있다.
- **SC-010**: 교사는 수업 → 과제 → 학생 3단계 탐색으로 특정 학생의 전체 제출 이력(코드 포함)에 4번 이내 클릭으로 도달할 수 있다.
- **SC-011**: 과제 마감 후 교사가 유사도 분석을 실행하면 수강생 50명 기준 10분 이내에 결과가 제공된다.
- **SC-012**: 교사가 Publish를 실행하면 수강생 50명, 5문제, 최대 20회 시도(최대 5,000 커밋) 기준 전체 업로드가 **10분 이내**에 완료된다. GitHub Publish는 REST API가 아닌 git push 프로토콜을 사용하여 Rate Limit 제약을 받지 않는다.
- **SC-013**: Publish 완료 후 GitHub에서 특정 학생 브랜치의 `git log`를 통해 해당 학생의 모든 제출 시도 이력(코드 + 커밋 메시지)이 시도 순서대로 조회된다.

## Assumptions

- 지원 언어 초기 범위: Python 3, Java 17, C++17, C17 (추후 확장 가능).
- 모바일 웹 지원은 v1 범위 밖; 데스크탑 브라우저(Chrome, Firefox, Edge 최신 버전) 기준.
- 코드 에디터는 Monaco Editor(VS Code 기반) 수준 제공: 문법 강조, 자동 들여쓰기, 괄호 자동 완성. 파일 트리, 터미널, 다중 파일 관리 등 풀 IDE 기능은 v1 범위 밖.
- Google Workspace for Education 계정을 가진 교육기관을 주요 사용자로 가정.
- 교사는 Google Classroom에서 과제를 먼저 생성하고 이후 온라인 저지에서 문제 세트와 연동한다.
- 채점 방식은 출력 비교(output matching) 기준; 스페셜 저지(커스텀 채점자)는 v1 범위 밖.
- 한 문제에 대한 Accepted 이후 재제출은 시도 횟수에 포함되지 않으며 점수를 변경하지 않는다.
- 코드 실행 환경의 최대 시간 제한은 10초, 메모리 제한은 512MB.
- 문제 세트의 총 배점은 Google Classroom 과제의 배점과 동일하게 설정되며, 각 문제의 배점 합이 총 배점을 초과하지 않도록 교사에게 경고를 제공한다.
- 시도 횟수 구간이 겹치지 않도록 저장 시 유효성 검사를 수행한다.
- 코드 유사도 분석은 토큰 기반 비교 방식(변수명 정규화 후 구조 비교)을 사용; 외부 MOSS API 의존 없이 자체 구현.
- 제출 기록은 영구 보존; 별도 삭제 기능은 v1 범위 밖 (GDPR 삭제 요청은 관리자 수동 처리).
- 학생 풀이 과정에 대한 AI 피드백 기능은 v2에서 구현; v1에서는 기록만 축적.
- GitHub Publish 기능은 교사가 GitHub OAuth를 별도로 연동한 경우에만 사용 가능; 미연동 시 해당 기능 비활성화.
- Publish된 GitHub 저장소는 기본 private으로 생성; 교사가 수동으로 공개 전환 가능.
- 학생 브랜치 명칭에 사용하는 학생 식별자는 Google 이메일 로컬 파트를 슬러그화한 값(특수문자 → 하이픈)으로 결정.
- GitHub Publish는 REST API가 아닌 **git push 프로토콜**(PyGit2 또는 subprocess git)을 사용하여 5,000 커밋을 Rate Limit 없이 업로드한다. REST API 방식은 50명 × 5문제 × 20시도 = 5,000 커밋 × 5 호출 = 25,000 요청으로 GitHub 한도(5,000회/시간)를 5배 초과하므로 사용 불가.
- 1회 수업 최대 부하(5,000건/60분, 피크 2.8건/초)를 처리하기 위해 Celery 워커 8개 병렬 운영. 채점 대기 시간은 피크 시 최대 5분으로 설계.
- 유사도 분석 임계값 기본값은 80%이며 교사가 과제별로 조정 가능.
